# AGENTS.md

## 适用范围（Scope）

- 适用于当前目录下的项目文档（project documentation），包括需求、设计、技术、接口、数据、测试、运维、安全、发布、研究和协作文档。
- 本文件在 `docs/` 目录采用一套可复用的通用文档标准；迁移到其他项目时，应先替换为目标项目的事实来源、路径、命令和术语。
- 与父级 `AGENTS.md` 叠加生效；发生冲突时，更靠近当前文件的规则优先（本地优先 / local wins）。

## 定位（Context）

- 本文件是面向 agent 的文档编写与审查规范，不是项目背景资料全集；只指导如何选择、编写、维护和验证项目文档。
- 通用框架只提供结构和检查维度；涉及当前仓库或目标项目的产品规则、接口契约、数据口径、命令、路径和验收方式时，以真实文件和用户当前指令为事实来源（source of truth）。

## 文档工作原则

- 修改前先识别文档类型、目标读者、使用场景、当前状态和事实来源（source of truth）。
- 一份文档优先服务一个主要目的：学习、执行、查询、解释、决策、契约、验收、运行或追溯。
- 同一事实只维护在一个权威位置；其他文档用链接引用，避免复制后漂移。
- 稳定事实进入长期文档；会议流水、临时计划、探索草稿和一次性排期不要覆盖正式文档。
- 明确区分事实、决策、假设、示例、待确认问题和历史记录；未确认内容标注 `待确认` 或 `假设`，集中列表统一命名为 `Open questions`。
- 不伪造数据、调研结论、竞品信息、技术约束、接口行为、测试结果、执行结果或来源。
- 引用路径、命令、接口、表名、配置项、版本号或外部链接时，先验证其存在；无法验证时明确说明。

## 文档类型选择

- 以下名称用于统一文档类型和职责，不强制文件名；引用已有文档时，必须使用真实标题、真实路径和当前仓库的链接格式。
- 内容回答“用户为什么需要、能做什么、受什么产品规则约束”时，写入 `项目需求文档（PRD）`。
- 内容回答“用户如何看见、理解、操作和获得反馈”时，写入 `用户体验与界面设计规范（UX/UI）`。
- 内容回答“系统如何拆分、协作、部署、扩展和失败恢复”时，写入 `技术方案（Architecture）`。
- 内容回答“为什么选择这个方案、放弃哪些选项、影响是什么”时，写入 `架构决策记录（ADR / Decision Record）`。
- 内容回答“数据事实如何表达、约束、迁移和保留”时，写入 `数据模型（Data Model）`。
- 内容回答“调用方如何集成、输入输出是什么、错误如何处理”时，写入 `接口文档（API Spec）`。
- 内容回答“如何证明需求被满足、风险被覆盖”时，写入 `测试方案（QA Plan）`。
- 内容回答“线上如何监控、排障、回滚、升级和复盘”时，写入 `运维手册（Runbook）`。
- 内容回答“权限、隐私、审计、合规和响应责任是什么”时，写入 `安全与合规（Security & Compliance）`。
- 内容回答“我们知道什么、凭什么知道、结论适用边界是什么”时，写入 `调研与分析（Research）`。
- 内容回答“读者如何完成一个具体任务”时，写入 `操作指南（How-to）`；回答“概念为什么这样设计”时，写入 `概念解释（Explanation）`；回答“字段、命令、配置等事实清单”时，写入 `参考清单（Reference）`。

## 文件命名建议（Filename Template）

- 文件命名是建议（recommended），不强制为符合模板而重命名已有文件；新增、迁移或主动整理文档时，优先采用本节模板。
- 推荐模板：`<中文主题或对象> <English anchor> <中文文档类型>.md`；无独立主题时，优先使用 `<中文文档类型> <English anchor>.md`。
- 行业缩写或强入口词可以前置，例如 `PRD 项目需求文档.md`、`UX UI 设计规范.md`、`API Spec 接口文档.md`、`ADR 架构决策记录.md`；描述型英文通常后置，例如 `数据模型 Data Model.md`、`技术方案 Architecture.md`。
- 中文可读优先：文件名必须让中文读者直接判断文档职责；英文锚点只保留行业通用、搜索友好或能减少歧义的关键词，例如 `PRD`、`UX UI`、`API Spec`、`Data Model`、`QA Plan`。
- 文件名里的英文锚点使用空格分词，不使用 `/`、`\`、`:` 或括号；正文标题可以使用括号形式，例如 `用户体验与界面设计规范（UX/UI）`。
- 文件名通常不重复仓库名或项目名；标题、摘要或元信息里再写完整项目名称。
- 推荐文件名示例：`PRD 项目需求文档.md`、`UX UI 设计规范.md`、`技术方案 Architecture.md`、`ADR 架构决策记录.md`、`数据模型 Data Model.md`、`API Spec 接口文档.md`、`测试方案 QA Plan.md`、`运维手册 Runbook.md`、`安全与合规 Security Compliance.md`、`调研与分析 Research.md`、`术语表 Glossary.md`。
- 面向单一专题的文档，在前面加入中文主题：例如 `会员权限 API Spec 接口文档.md`、`奖品兑换 QA Plan 测试方案.md`、`登录故障 Runbook 运维手册.md`。
- 引用文档时以真实文件名为准；如果展示名、标题和文件名不一致，优先修正文档链接和交付说明，不要凭模板猜路径。

## 通用写作与审查

- 中文为主，必要英文作为关键词锚点（English keyword anchor）；同一概念首次出现时优先写成 `中文名（English anchor）`，后文优先使用中文名。
- 正式文档类型使用本文约定的中文名称；英文缩写或英文名只作辅助锚点，除 `README`、`ADR`、`API` 等行业通用写法外，不单独替代中文名称。
- 展示名和文件名可以不同；展示名不得被当作路径使用，真实引用必须匹配仓库内实际文件名。
- 同一概念固定一个名称：验收标准（acceptance criteria）、范围外事项（out of scope）、破坏性变更（breaking change）、主路径（happy path）、重要变更（notable changes）、待确认问题（Open questions）。
- 新增重要文档时，优先交代：标题、状态、读者、目标、范围、事实来源、正文、验证或使用方式、待确认问题（Open questions）。
- 只采用服务当前文档目标的章节和检查项；不要为了套用通用标准而扩写无关背景或保留空模板。
- 使用短段落、列表和表格提升可扫读性；表格只在能减少歧义时使用。
- 标题应反映内容任务：操作步骤用动词短语，概念解释用名词短语，参考清单用对象名称。
- 示例必须贴近真实项目，能运行或能被读者映射到真实场景；不要用看似完整但无法验证的伪数据撑版面。
- 重要规则应写成可判断语句，避免“友好、合理、完善、正常”等不可验收表达。
- 修改文档时同步检查相邻文档是否需要更新，尤其是用户可见行为、接口契约、数据口径、安全规则、测试口径、发布说明和运维步骤。
- 删除或迁移内容时，确认是否存在依赖；必要时留下链接、迁移说明或废弃原因，除非明确要求。
- 不要为了模板完整保留空章节；当前文档不需要的章节应删除。

## 项目需求文档（PRD）

项目需求文档（PRD）回答“为什么做、为谁做、做什么、做到什么程度算完成”。它应给产品、设计、研发、测试和业务干系人形成共享理解，同时不替工程做实现决策。

- 应包含：背景与问题、目标用户与场景、产品目标、成功指标（success metrics）、本期范围、范围外事项（out of scope）、核心流程、分支与异常流程、功能需求、业务规则、角色与权限差异、关键状态、体验边界、数据与分析需求、依赖、风险、待确认问题（Open questions）、验收标准（acceptance criteria）。
- 功能需求应描述用户可见行为：入口、页面或对象、用户动作、系统反馈、状态变化、错误处理、空状态、加载、权限不足、重复提交、弱网或中断后的结果。
- 业务规则应描述产品不变量（business invariants），例如额度、次数、审核、撤销、可见性、时间窗口、状态流转和边界条件；不要指定用哪张表、哪个函数或哪种事务实现。
- 数据与分析需求只写产品需要展示、记录或分析的指标和口径；数据库字段、索引、读模型和存储生命周期应写入数据模型（Data Model）。
- 验收标准应可测试，可使用 `Given / When / Then`，并覆盖主路径、关键异常、权限差异和不可做的事。
- 不应包含：数据库设计、API 路径、云函数或服务拆分、类名方法名、缓存队列、锁、幂等 token、部署方案、环境变量、密钥、研发任务拆分、工时估算、完整测试用例或视觉像素稿。
- 审查项目需求文档时重点确认：问题是否真实清楚，范围是否闭合，规则是否无冲突，异常状态是否可处理，验收是否可验证，是否混入技术实现，是否遗漏相关设计规范、接口文档、数据模型或测试方案链接。

## 用户体验与界面设计规范（UX/UI）

用户体验与界面设计规范（UX/UI）回答“用户如何理解和操作”。它应把产品规则转成具体体验，不替代项目需求文档的范围定义，也不替代前端实现方案。

- 应包含：信息架构、导航关系、页面状态、组件使用规则、交互反馈、视觉 token、文案语气、空/加载/错误/权限状态、响应式或多端适配、可访问性要求。
- 组件规范应写清使用场景、结构、变体、状态、禁用条件、错误提示、内容长度、触控区域、键盘/读屏等辅助交互要求。
- 视觉规范应描述颜色、字体、间距、图标、动效和品牌语气的规则，不要只贴静态截图。
- 审查时确认设计是否覆盖主流程和异常流程，文案是否与项目需求文档术语一致，可访问性是否有可检查标准，组件状态是否足够工程实现和测试验收。

## 技术方案（Architecture）

技术方案（Architecture）回答“系统如何实现并长期演进”。它应解释关键取舍、系统边界和质量属性，不复制代码，也不替代接口文档（API Spec）、数据模型（Data Model）或运维手册（Runbook）。

- 应包含：背景、目标与非目标、约束、上下文边界、模块职责、数据流、运行时流程、外部依赖、权限模型、错误处理、事务或一致性策略、性能与扩展性、可观测性、迁移与回滚、风险和验证方式。
- 对架构视图保持足够但不过量：上下文图说明外部关系，模块图说明静态拆分，流程图说明关键运行时交互，部署图说明环境和依赖。
- 关键方案应记录备选项和取舍原因；难以逆转或影响质量属性的决定应拆到架构决策记录（ADR）。
- 审查时确认方案是否满足项目需求文档和非功能需求，边界是否清楚，失败场景是否被处理，数据一致性和权限是否闭合，迁移/回滚是否可执行。

## 架构决策记录（ADR / Decision Record）

架构决策记录（ADR）记录一个重要且较难逆转的决策，重点回答“为什么是这个选择”。

- 应包含：标题、状态、背景（context）、决策驱动因素、备选方案、决定（decision）、影响（consequences）、关联文档和日期。
- 每篇架构决策记录只记录一个决策；不要把方案全文、会议纪要或事后争论塞进架构决策记录。
- 审查时确认决策是否真的具有架构或长期影响，是否写清被放弃选项，影响是否包含成本、风险和后续约束。

## 数据模型（Data Model）

数据模型（Data Model）回答“数据事实如何表达、约束和维护”。它是数据口径和持久化约束的事实来源。

- 应包含：实体与关系、字段语义、类型、必填性、唯一性、索引、枚举、状态机、约束、默认值、派生字段、读写来源、迁移、回填、归档、删除和保留策略。
- 写清数据口径和业务规则的映射，例如一个状态由谁写入、何时变化、是否允许回滚、历史记录如何保留。
- 不要重复项目需求文档的完整用户叙事；只保留理解数据语义必需的业务上下文。
- 审查时确认字段是否有明确语义，状态是否无歧义，约束是否能支撑产品规则，迁移是否包含兼容、回填和失败处理。

## 接口文档（API Spec）

接口文档（API Spec）回答“调用方如何可靠集成”。契约应稳定、可验证、可被工具或测试复用。

- 应包含：版本、base URL 或调用入口、鉴权、权限、路径/方法、请求参数、请求体、响应体、错误码、幂等、分页、排序、过滤、限流、兼容性、示例和废弃策略。
- 示例应覆盖成功、参数错误、权限不足、资源不存在、冲突、重复提交和服务端失败等关键场景。
- 结构化接口优先维护 OpenAPI、JSON Schema、protobuf 或等价契约；Markdown 说明应引用契约而不是另写一份容易漂移的接口。
- 审查时确认参数和响应是否完整，错误语义是否稳定，是否泄露内部实现，示例是否可运行，破坏性变更（breaking change）是否有迁移说明。

## 测试方案（QA Plan）

测试方案（QA Plan）回答“如何证明需求被满足”。它应连接需求、风险、用例和验收证据。

- 应包含：测试目标、范围、不测范围、需求追踪、风险矩阵、测试环境、测试数据、关键用例、边界条件、回归范围、自动化策略、进入/退出条件和未覆盖风险。
- 测试用例应写清前置条件、步骤、预期结果和数据要求；高风险流程要覆盖失败、重试、权限、并发、兼容和回滚。
- 测试记录和缺陷流水不要替代测试方案；需要时链接到执行记录或缺陷系统。
- 审查时确认每个关键需求都有验证路径，验收标准可被执行，用例没有只覆盖主路径（happy path），剩余风险被明确说明。

## 运维手册（Runbook）

运维手册（Runbook）回答“线上出事或变更时如何执行”。读者可能在压力下使用，因此步骤必须直接、具体、可回滚。

- 应包含：适用场景、告警信号、影响范围、前置权限、诊断步骤、缓解措施、修复步骤、回滚方式、验证方式、升级路径（escalation）、沟通模板和复盘入口。
- 命令、面板、日志查询、配置路径和开关名称必须真实可用；危险操作前写清确认条件和预期影响。
- 审查时确认新人能否按步骤执行，失败时是否有下一步，恢复后是否有验证和复盘动作。

## 安全与合规（Security & Compliance）

安全与合规（Security & Compliance）回答“哪些行为被允许、受限、审计或响应”。它应写规则和责任，不暴露可被滥用的细节。

- 应包含：数据分级、权限模型、敏感信息处理、隐私规则、审计日志、密钥管理原则、依赖风险、威胁场景、合规要求、响应流程和责任人角色。
- 不写入密钥、token、个人账号、生产凭据、绕过方式或未脱敏的个人信息。
- 审查时确认规则是否可执行，最小权限是否明确，日志和留存是否符合要求，安全例外是否有期限和负责人。

## 调研与分析（Research）

调研与分析（Research）回答“我们知道什么，以及凭什么知道”。它为决策提供证据，不直接替代项目需求文档或技术方案。

- 应包含：研究问题、方法、样本、时间、数据来源、证据、洞察、限制、置信度、结论和对决策的影响。
- 竞品分析要区分事实观察、推测原因和可借鉴点；用户研究要保留样本限制，不把个案写成普遍结论。
- 审查时确认结论是否由证据支撑，数据是否可追溯，限制是否明确，是否有具体决策影响。

## 快速入口、操作指南、参考清单与术语表（README / How-to / Reference / Glossary）

- 快速入口（README）面向快速理解和上手，应包含项目用途、入口、安装运行、常用命令、目录导航、验证方式和下一步链接；不要复制全部架构或需求背景。
- 操作指南（How-to）面向完成具体任务，应写前置条件、步骤、预期结果、失败处理和验证方式；不要混入长篇概念解释。
- 参考清单（Reference）面向查询事实，应保持完整、准确、稳定和结构一致，例如配置项、命令、字段、错误码或术语表。
- 术语表（Glossary）只收录项目内有稳定含义的术语；同义词、旧名称和歧义名称应给出映射或废弃说明。

## 变更日志与发布说明（Changelog / Release Notes）

- 变更日志（Changelog）面向维护者和集成方，按版本或日期记录重要变更（notable changes）；发布说明（Release Notes）面向用户或业务方，强调影响、价值和迁移动作。
- 推荐分类：`Added`、`Changed`、`Deprecated`、`Removed`、`Fixed`、`Security`；没有内容的分类不要保留空标题。
- 不直接粘贴提交日志（commit log）；把内部改动翻译成读者能理解的影响，并标出破坏性变更（breaking change）、迁移步骤和安全更新。
- 审查时确认版本、日期、影响范围和迁移说明准确，和实际发布内容一致。

## 最终验证（Validation）

- 检查文档类型是否正确，是否混入其他文档职责。
- 检查新增或迁移文档的文件名是否符合建议模板；若不符合，是否有清楚理由。
- 检查读者、目标、范围、事实来源、术语和状态是否清楚。
- 检查产品规则、接口契约、数据口径、测试口径、安全要求和运维步骤是否互相一致。
- 检查验收标准（acceptance criteria）、API 示例、操作步骤、测试用例或回滚步骤是否可验证。
- 检查是否存在未标注的假设、过期链接、无法验证的口号、重复维护的事实或敏感信息。
- 若引用外部最佳实践或行业资料，按下方参考框架使用边界处理。

## 外部参考框架使用边界（External References）

- 以下框架只作为结构参考，不替代当前仓库或目标项目的事实来源；使用前先确认它适合当前文档类型、读者和交付目标。
- 引用外部最佳实践时，优先使用权威、稳定、近期仍适用的来源，并在交付说明中列出关键来源和适用边界。
- Diataxis：区分 `Tutorial`、`How-to`、`Reference` 和 `Explanation`。
- Atlassian / Aha：项目需求文档（PRD）应围绕目标、用户、功能、范围和成功标准形成共享理解。
- arc42：技术方案可从目标、约束、上下文、构建块、运行时、部署、质量和风险组织。
- ADR / MADR：用 `Context`、`Decision`、`Consequences` 记录重要决策。
- OpenAPI：结构化维护 HTTP API 契约。
- Google SRE：运维手册（Runbook）和事件响应（incident response）应强调可执行、可升级、可复盘。
- Keep a Changelog：发布变化应按读者关心的重要变更（notable changes）组织。
